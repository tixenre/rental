Lógica de gating del Confirmar
/* El CTA depende de tres puertas, en orden de prioridad */
const empty     = items.length === 0;
const noDates    = !range;
const conflicts  = items.filter(i => !hasStock(i, range));   // sobrevenido

const cta =
  empty        ? null                       // no hay drawer footer, hay EmptyState
: noDates       ? { label: "Elegí fechas primero", disabled: true }
: conflicts.length ? { label: "Resolvé el conflicto", disabled: true }
: submitting    ? { label: "Enviando solicitud…", loading: true }
:                 { label: "Confirmar solicitud" };

/* El conflicto se re-chequea al abrir el drawer y al cambiar fechas
   (useQuery dias-bloqueados) — el stock pudo cambiar desde que se agregó. */
Cuatro estados, una sola puerta visible: el footer muestra el bloqueo de mayor prioridad — vacío > sin fechas > conflicto > enviando > ok. Nunca dos barras a la vez. · Vacío reemplaza toda la lista por EmptyState con CTA que cierra el drawer y lleva al catálogo — sin footer ni totales. · Sin fechas: date pill en dashed --naranja, precios de línea en placeholder muted ($ X / día, no total), total — elegí fechas, CTA disabled. El precio existe pero no se puede totalizar sin período. · Conflicto sobrevenido: el item culpable va con borde destructive + barra inline que nombra el día y ofrece cambiar fechas; el footer resume cuántos equipos fallan. Se re-chequea al abrir y al cambiar fechas — el stock pudo moverse. · Confirmando: items y date pill atenuados (opacity .55, sin interacción), close deshabilitado, CTA con spinner y cursor: progress. Al volver: toast success + navegación al portal. · El total en confirmando no se atenúa — el usuario quiere verlo mientras espera.
